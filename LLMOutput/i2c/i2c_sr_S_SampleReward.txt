It reflects how the various blocks (shown in Fig. 5.1 and the state diagram) work together to carry out I2C transactions.\n### 1. WISHBONE Interface\n- **Purpose**: Acts as the main bus interface for the core. It provides signals such as `wb_clk_i`, `wb_rst_i`, address (`wb_adr_i`), data input (`wb_dat_i`), data output (`wb_dat_o`), and control signals (`wb_we_i`, `wb_stb_i`, `wb_cyc_i`, `wb_ack_o`, `wb_inta_o`).\n- **Operation**:  \n  - The CPU or system master writes to and reads from the I2C core’s registers (Prescale, Control, Command, Transmit, Receive, and Status) through this interface.  \n  - All output signals from the core are registered, and each bus access takes 2 clock cycles.\n### 2. Prescale Register & Clock Generator\n- **Prescale Register**:  \n  - Holds an integer value used to divide down the incoming master clock (`wb_clk_i`) to generate the internal SCL frequency (5× the target SCL).  \n  - Must be programmed when the core is disabled (`EN = 0` in the Control register).  \n- **Clock Generator**:  \n  - Produces the internal 4× or 5× SCL clock enable signals that drive the Bit Command Controller.  \n  - Manages clock stretching if a slave holds SCL low, ensuring correct synchronization on the I2C bus.\n### 3. Control & Status Registers\n- **Control Register** (`CTR` at address 0x02):  \n  - Enables or disables (`EN` bit) the I2C core.  \n  - Enables or disables (`IEN` bit) the core’s interrupt mechanism.  \n- **Status Register** (`SR` read-only at address 0x04):  \n  - Reflects real-time conditions such as `RxACK` (received ACK), `Busy` (bus busy), `AL` (arbitration lost), `TIP` (transfer in progress), and `IF` (interrupt flag).\n### 4. Command Register & Byte Command Controller\n- **Command Register** (`CR` write-only at address 0x04):  \n  - Bits like `STA` (start), `STO` (stop), `RD` (read), `WR` (write), and `ACK` (acknowledge control) drive high-level I2C actions.  \n  - The Byte Command Controller interprets these bits and breaks them down into individual bit-level operations.\n- **Byte Command Controller**:  \n  - Works at the byte level, orchestrating sequences like:  \n    1. Generating START (or repeated START).  \n    2. Writing/Reading a byte to/from the slave.  \n    3. Checking or sending ACK/NACK.  \n    4. Generating STOP if required.  \n  - Transitions between states (Idle → START → WRITE or READ → ACK → STOP) based on the command bits.\n### 5.
